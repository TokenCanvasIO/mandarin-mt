## Audit: concrete bugs

### Verified claims (your six checks)

**(1) `runStep` / `pauseBar` / `reveal` audio sequencing**

Within a **single** `runStep()` call, the auto-play path does await in order (`speak` → `pauseBar` → `jingtao` → reveal) in `renderDrill` (lines 798–804) and `renderExchange` (lines 774–778).

However, steps **can overlap** in practice:

- **`advance()`** (line 844–847) calls `runStep()` with no `await` and no re-entry guard; rapid double-clicks on `next →` / `continue →` can run multiple `runStep()` instances in parallel, each clearing `#stage` and driving audio.
- **`Audio.jingtao`** (line 641) always schedules `setTimeout(()=>res(true), 8000)`, so the promise can resolve **before** playback ends; `renderDrill` then continues to reveal / Harold note while Mandarin audio may still be playing.
- **`Audio.speak`** (line 614) has the same early-resolve timeout pattern.

**(2) “First view cannot skip pause”**

Works **only in-memory, same session, first render of that step**:

- `firstView = !viewed.has(...)` (line 727) is evaluated **before** `viewed.add` (line 735), so the first in-session render correctly passes `pauseBar(false)` via `pauseBar(!firstView)` (lines 774, 799).
- `viewed` is **not persisted** (line 581); after reload/resume, every step is treated as first view → pause is **never skippable** on return visits.
- `tones` / `wordorder` early-return (lines 729–730) skips `viewed.add`, so those steps are always “first view” if ever re-entered.

**(3) `welcomeBack` / `build(['ni'])` crash**

**No crash.** `build(['ni'])` is valid (`ni` exists in `SYL`).

Problems are logic, not runtime:

- `['nihao'].map ? ['ni'] : ['ni']` (line 959): `Array.prototype.map` is always truthy → always `['ni']`; dead/confusing code. Assigned to unused `v`.
- `lastId = L.intro[state.si]` (line 958) indexes the lesson **vocab intro list** by **step index** — wrong mapping once `state.si` exceeds `intro.length` or passes intro-only steps. Shows misleading “last word” (e.g. mid–Lesson 3 shows an early intro word, not the last drill vocabulary).

**(4) `logWords` double-count / mis-attribution**

**Yes, multiple concrete issues** in `logWords` (lines 680–690):

- **Re-count on resume:** `welcomeBack` → `continue →` `runStep()` re-renders the saved step and calls `logWords` again → `seenCount++` and `promptCounter++` for the same step.
- **Exchange steps never log words:** `logWords(step)` uses `step.syl||[]`; `EX(...)` steps have no top-level `syl` (only `lines[].syl`) → `chars` is `''` → no `mark()`, but `promptCounter` still increments.
- **Substring false positives:** `chars.includes(wc)` marks shorter vocab inside longer strings (e.g. drill `nihao` (你好) also marks `hao` (好); any compound containing 人 marks `ren`).

**(5) Azure fetch error handling**

No throw/crash — `azure()` catches and returns `null` (lines 620–628). Failures are **silent**; `jingtao` falls back to Google/Youdao with no indication Azure failed. `testSet` (line 943) does not surface success/failure. **Degraded behaviour, not a runtime error.**

**(6) `LESSONS[7]` patch + “May I go?” ternary**

**Not a functional bug.** Ternary `[keSandhi(),'yi3','wo'].length ? …` is always truthy (length 3), so `D()` already gets `['wo', keSandhi(), 'yi3', 'qu']`. Post-init patch (line 555) is **redundant** but harmless. Result is correct 我可以去.
